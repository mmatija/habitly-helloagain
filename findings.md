# Findings

## Running the app with `docker compose up`

- Right off the bat, I wasn't even able to run the `docker compose up` command successfully because `mysql` image was not compatible with ARM architecture. I added an explicit `platform` attribute to `db` service, after which I was able to run the containers.
- Next thing I noticed was that there are a lot of errors coming from both backend and frontend apps. I decided to look at the backend first because it might be a reason why fronted does not work. I noticed that the backend service complains about `SECRET_KEY` environment variable missing. I noticed that all environment variables are given in [.env.example](backend/.env.example) file and that the point was to create a local `.env` file, but I found this conflicting with instruction that the app should start cleanly by running `docker compose up` so I decided to add missing variable to the `docker-compose.yml` file instead. After this change, **the backend app was starting cleanly without any errors**.
- This got the backend to run eventually, but it kept crashlooping the first few seconds because it could not connect to the `mysql` database. The reason was that it was trying to make the connection before the database was fully initialized. This did not align well with the requirement that the app should start cleanly by running `docker compose up`, so I added a simple retry mechanism to the backend app, which tries to connect to the database every 5 seconds until it succeeds. **At this point the backend was starting cleanly** without any errors. 
- The remaining errors were coming from the frontend app, which was failing due to mismatch between node version used in the frontend app and the one specified in the `Dockerfile`. I updated the `Dockerfile` to use node 16, which is the version specified in [Github Workflows](.github/workflows/build.yml) file. After this change, **the frontend app was also starting cleanly without any errors**.


## Exploring the state of current login
- After I got the app to start cleanly, I wanted to explore the state of current login to see if any part of it can be salvaged as-is. Upon further exploration and reading both the provided instructions, as well as the [Readme](README.md) file, I realized that in order for the redirect to `localhost` to work, I would need to have access to the existing Auth0 tenant, which I do not have. The suggestion from the author is to create my own Auth0 tenant and update the configuration, but that would also mean that everyone else who wants to run the app would have to do the same thing. This again does not align well with the requirement that the app should run cleanly by running `docker compose up`, so **I decided to proceed with custom authentication mechanism**.

## Implementing custom authentication mechanism
- For authentication mechanism, I decided to **stick with the one that most closely resembles the existing one**, in hopes that it would simplify the replacement. That means I went for JWT based authentication which uses RS256 algorithm as it is the one used by Auth0.
- The approach was to introduce new login mechanism **in parallel with the existing one**, which would allow me to move in smaller steps and have a reference point to compare against. I started by adding a new login endpoint in the backend, which would accept username and password and return a JWT token if the credentials are valid. I also added a new login page in the frontend, which would allow users to enter their username and password and log in using the new login endpoint.
- Once I had the login flow working, it was time to replace the authentication for existing  endpoints with the new one. This was a bit tricky because the existing endpoints were tightly coupled with the Auth0 authentication mechanism - for the sake of simplicity, I decided to replace the authentication for all endpoints at once, which one would probably not do in the real world scenario. This turned out to be surprisingly smooth and I did not encounter any major issues during this process. After this change, **the app was fully functional with the new authentication mechanism**.

## Further improvements
- The current implementation of authentication is very basic and does not include features like password reset, token revocation, etc. In the real world scenario, these features would have to be implemented as well, but for the sake of this task, I decided to focus on the core authentication mechanism and leave these features for later.
- As I added some tests, it would make sense to update the README file with instructions on how to run them, as well as the general documentation as it is not outdated (no more Auth0 for example)
- Frameworks and libraries used are outdated to the point where it was a hindrance - I wanted to use some libraries but couldn't due to version incompatiblity. It would make sense to gradually update the dependencies to newer versions.
